## 背景与优化说明

本手册是对之前宏观规划的**落地级细化**。基于你的反馈，做了以下关键优化：

1. **目录结构重新设计**：放弃示例级的 `Reception/Backend` 扁平结构，采用更适合私有化部署的 **`backend/` + `frontend/` 分离结构**，内部仍按功能模块垂直切分。
2. **镜像提前规划**：明确拆分出 7 个核心镜像（`navima-api`、`navima-worker`、`navima-sandbox`、`navima-web`、`navima-admin`、`navima-license-api`、`navima-license-web`），为 Helm/Docker 部署打好基础。
3. **开发路径线性化**：不再使用 P0/P1/P2/P3 阶段划分，而是按**严格依赖顺序**排列 25 个模块。每个模块细到"下一步该改哪个文件"。
4. **零硬编码**：所有域名、品牌名、超时时间全部走环境变量。

| `navima-worker` | `deploy/build/worker/Dockerfile` | `backend/worker/` | Celery | 异步任务 |

### 1.3 关键目录规则

- **禁止硬编码**：所有配置走 `backend/api/app/core/config.py`（Pydantic Settings）→ `.env` → ConfigMap。
- **模块自治**：每个 `modules/{name}/` 包含 `routes.py`、`schemas.py`、`services.py`、`models.py`、`dependencies.py`。
- **前端原子化**：改登录按钮，直接去 `frontend/web/src/modules/auth/login/components/login-button/index.tsx`。

### 1.4 衔接层（link）设计规范

`frontend/*/src/modules/link/` 是前端模块之间的"胶水层"，禁止包含任何业务逻辑，只做以下事情：

| 文件 | 职责 | 引用关系 |
|---|---|---|
| `AppRouter.tsx` | 注册所有 feature modules 的路由，使用 `React.lazy()` 做路由级代码分割 | 只 import 各模块的 `pages/*` |
| `AuthGuard.tsx` | 全局登录状态拦截，未登录跳转 `/login` | 只读取 shared 的 auth store |
| `WorkspaceProvider.tsx` | 挂载当前 workspace 上下文 | 只调用 `workspaceService.getCurrent()` |
| `MainLayout.tsx` | 侧边栏菜单 + 顶部导航 | 菜单配置数组硬编码在此文件（标题从 i18n 读取） |
| `PortalRouter.tsx` | Chat Portal 独立路由 | 同 `AppRouter` 规则 |

**关键约束**：`link` 层可以 import 任何 feature module 的 `pages/` 和 `types/`，但**禁止** import `components/`、`services/`、`hooks/` 等内部实现，防止循环依赖。

**路由注册与菜单管理规范**

每新增一个前端页面模块，必须完成以下两步（文档中不再逐模块重复，作为统一规范执行）：

1. **路由注册**：在 `frontend/web/src/modules/link/AppRouter.tsx`（或 `frontend/admin/src/modules/link/AppRouter.tsx`）中使用 `React.lazy()` 注册该模块的 `pages/*` 路由。示例：
   ```tsx
   const NodesPage = React.lazy(() => import('../dashboard/nodes/pages/NodesPage'));
   ```
2. **菜单更新**：在 `MainLayout.tsx` 的菜单配置数组中添加对应菜单项（标题从 `i18n` 读取，图标使用 Ant Design icon）。示例：
   ```tsx
   { key: 'nodes', label: t('menu.nodes'), icon: <ClusterOutlined />, path: '/dashboard/nodes' }
   ```

**强制执行**：在 `.eslintrc.cjs` 中配置 `no-restricted-imports` 规则：
```js
{
  files: ['frontend/*/src/modules/link/**/*'],
  rules: {
    'no-restricted-imports': ['error', {
      patterns: [
        { group: ['*/modules/*/components/**', '*/modules/*/services/**', '*/modules/*/hooks/**'], message: 'link 层禁止导入 feature module 内部实现' }
      ]
    }]
  }
}
```

## 2. 一步一步的模块开发计划（线性依赖顺序）

以下 25 个模块必须**按顺序逐一开发**。每个模块都包含：
- 该模块依赖什么（必须等前面的模块完成）
- 精确到文件名的目录树
- 数据库表 DDL
- API 路由清单
- 前端页面/组件清单
- 核心交互流程（分步骤）

#### 关键文件内容

- `docker/docker-compose.yaml`：**仅启动基础设施服务**：`postgres:16`、`redis:7`、`minio:latest`。不包含 `api`、`worker`、`web`、`admin`。
- `.env.example`：包含 `APP_ENV`、`APP_DOMAIN`、`ADMIN_DOMAIN`、`DB_*`、`REDIS_*`、`SMTP_*`、`LICENSE_*` 模板，所有值留空。
- `backend/api/app/main.py`：空的 FastAPI 应用，提供两个健康检查端点：
  - `GET /health/live`（Liveness）：进程是否存活，快速返回 `{"status":"ok"}`
  - `GET /health/ready`（Readiness）：DB 和 Redis 是否可连接，用于 Kubernetes 流量切入
  - 开发环境默认暴露 `/docs`（Swagger UI）和 `/redoc`，生产环境通过 `ENABLE_API_DOCS=false` 关闭
- `deploy/build/*/Dockerfile`：各镜像构建文件，与源码目录完全分离。Helm Chart 中配置 `livenessProbe` 指向 `/health/live`，`readinessProbe` 指向 `/health/ready`。
- `deploy/build/web/nginx.conf` 与 `deploy/build/admin/nginx.conf`：Nginx 配置文件，必须包含：
  - SPA fallback：`try_files $uri $uri/ /index.html;`
  - gzip 压缩开启
  - 静态资源缓存头（`Cache-Control: public, max-age=31536000`）
  - `/api/` 路径反向代理到 `navima-api` 服务（Docker Compose 中通过服务名解析）
- `.github/workflows/ci.yaml`：
  - 后端：运行 `pytest backend/api/tests` 和 `pytest backend/worker/tests`
  - 前端：运行 `npm run test`（Vitest）
  - 镜像：对 `deploy/build/` 下的每个 Dockerfile 执行 `docker build` 检查
  - 代码质量：运行 `ruff check backend/` 和 `tsc --noEmit`（前端类型检查）

#### 开发模式说明

- **后端开发**：进入 `backend/api/`，执行 `python -m app.main`（或 `uvicorn app.main:app --reload`），直接源码启动。
- **Worker 开发**：进入 `backend/worker/`，执行 `celery -A app.main worker --loglevel=info`，直接源码启动。
- **前端开发**：进入 `frontend/web/`，执行 `npm run dev`；进入 `frontend/admin/`，执行 `npm run dev`。
- **数据库/缓存**：执行 `docker compose -f docker/docker-compose.yaml up -d` 启动 PostgreSQL 和 Redis。

#### 技术栈与工程规范（Module 1 必须确定）

| 层级 | 选型 | 说明 |
|---|---|---|
| 后端框架 | FastAPI + Pydantic v2 | 统一使用 `async def` 路由 |
| 数据库 ORM | SQLAlchemy 2.0 (async) | 配合 `asyncpg` |
| 任务队列 | Celery + Redis | Worker 专用 |
| 前端框架 | React 18 + TypeScript + Vite | 前台和后台共用相同技术栈 |
| 状态管理 | Zustand | 全局状态（auth、workspace、theme） |
| 数据获取 | TanStack Query (React Query) v5 | 所有服务端状态统一用 Query，禁止裸 fetch |
| 表单处理 | React Hook Form + Zod | 所有表单强制使用 |
| UI 组件库 | Ant Design 5.x | 统一设计系统，禁止混用其他组件库 |
| 图表 | ECharts | 监控面板专用 |
| i18n | react-i18next | 翻译文件放在 `i18n/{lang}/{module}.json` |
| 测试后端 | pytest + httpx + pytest-asyncio | API 单元测试和集成测试 |
| 测试前端 | Vitest + React Testing Library | 组件和 hook 测试 |
| CSS | CSS Modules + CSS Variables | 禁止 CSS-in-JS（性能与 SSR 兼容性） |

**Alembic 迁移规范**（写入 `docs/migration-conventions.md`）：
- 命名格式：`{yyyy_mm_dd}_{HHMMSS}_{description}.py`
- 每个模块独立一个 migration 文件，禁止一个文件改多个模块的表（特殊情况除外）。
- 冲突解决：以 `alembic revision --autogenerate` 生成时间戳为准，后生成的 migration 在 `depends_on` 中显式依赖先生成的。

**数据初始化规范**：
- `scripts/setup/seed_data.py`：在首次部署时执行，插入默认值（`auth_settings`、空 `brand_configs`、初始 `licenses` 记录、默认管理员）。
- 每个模块的 `routes.py` 启动时不得自动写入缺省数据，必须通过显式 seed 脚本。

**错误码规范**（写入 `docs/error-codes.md`）：
- 格式：`{MODULE}_{ACTION}_{RESULT}`，如 `AUTH_INVALID_CREDENTIALS`、`AUTH_ACCOUNT_LOCKED`、`RATE_LIMIT_EXCEEDED`。
- 所有错误响应统一为：`{ "code": "...", "message": "...", "detail": {} }`。

#### 交互流程

1. 执行 `docker compose -f docker/docker-compose.yaml up -d`，确认 Postgres 和 Redis 已启动。
2. 进入 `backend/api/`，执行 `uvicorn app.main:app --reload --port 8000`。
3. 浏览器访问 `http://localhost:8000/health`，看到 `ok`。
4. 基建完成。

#### 数据库配置

- `backend/shared/navima_shared/db/session.py`：创建 `async_sessionmaker` 工厂。通过 `core/config.py` 读取 `DATABASE_URL`，并显式传入连接池参数 `pool_size`、`max_overflow`、`pool_timeout`。
- `backend/shared/navima_shared/db/base.py`：`DeclarativeBase` 基类，所有模型继承此基类。
- **共享策略**：`backend/api/` 和 `backend/worker/` 的 `pyproject.toml` 均通过本地路径依赖引用 `backend/shared/`（`{ path = "../shared", develop = true }`）。Worker 任务直接 `from navima_shared.db.models import Node, Deployment`。

#### 交互流程

1. 配置 `.env` 中的 `DATABASE_URL`。
2. 运行 `alembic revision --autogenerate -m "init"` 生成了一个空迁移。
3. 运行 `alembic upgrade head` 确认数据库连接正常。
4. 在 `main.py` 的 readiness health check 中增加 DB 连通性检测。

#### 文件说明

| 文件 | 职责 |
|---|---|
| `config.py` | `Settings(BaseSettings)`，读取 `.env`，字段包括：`app_env`, `app_domain`, `admin_domain`, `db_url`, `db_pool_size`, `db_max_overflow`, `db_pool_timeout`, `redis_url`, `secret_key`, `session_timeout_seconds`, `session_cookie_name`, `session_cookie_secure`, `session_cookie_samesite`, `session_redis_prefix`, `csrf_secret`, `rate_limit_redis_prefix`, `cors_origins`, `forward_connect_timeout`, `forward_read_timeout`, `forward_max_request_size`, `enable_api_docs`, `smtp_*`, `license_server_url`, `default_brand_name`, `default_logo_url`, `hf_endpoint` |

- `sglang_engine.py`：调用 `python -m sglang.launch_server`。

#### Sandbox（navima-sandbox）定位说明

`navima-sandbox` 是**安全隔离执行环境**，用于运行不受信任的代码或脚本：
- **核心职责**：执行用户上传的 Python 脚本（如自定义评估脚本、数据预处理脚本）、运行第三方模型 benchmark、执行动态 Adapter 代码。
- **安全边界**：通过 Linux Namespace + cgroups 限制 CPU/内存/网络，禁止访问宿主机文件系统（除显式挂载的只读目录）。
- **调用方式**：`navima-api` 通过 HTTP 调用 `navima-sandbox` 的 `/sandbox/run` 接口，传入代码和超时时间，沙箱执行后返回结果。
- **部署方式**：作为独立镜像运行，可与 `navima-api` 同节点但不同进程，生产环境建议部署在独立的低权限节点上。

#### Node Agent API 契约（取代裸 SSH）

为避免裸 SSH 的进程管理脆弱性，所有节点在纳管后必须运行 `scripts/agents/node-agent/app.py`（Ansible 自动下发）。Agent 暴露 HTTP API：

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/agent/process/start` | 接收 `{cmd, cwd, env, log_file}`，本地启动进程并返回 `pid` |
| POST | `/agent/process/stop` | 接收 `{pid}`，优雅停止进程 |
| GET | `/agent/process/{pid}/status` | 返回进程运行状态 |
| GET | `/agent/logs` | 返回指定日志文件尾部内容 |
| GET | `/agent/metrics` | 返回节点实时指标（供监控使用） |

**Agent 安全认证**：
- 每个节点首次探活时，`navima-api` 生成唯一的 `agent_secret`（32 字节随机字符串），写入 `nodes.agent_secret` 字段。
- Ansible 下发 Agent 时，将该 secret 注入 Agent 环境变量 `AGENT_SECRET`。
- 所有 Agent API 请求必须在 `Authorization: Bearer {agent_secret}` 头中携带该 secret。
- Agent 默认监听 `127.0.0.1:17321`（可通过环境变量覆盖），仅接受来自 API 服务 IP 的连接（配合节点防火墙规则）。
- `BaseEngine.start()` 通过 `nodes.agent_secret` 构造认证头，再调用目标节点的 `POST /agent/process/start`。

`BaseEngine.start()` 不再直接 SSH，而是由 Agent 负责本地进程生命周期管理（端口冲突检测、自动重启、日志轮转）。

#### 交互流程

1. 定义 `BaseEngine` 接口和异常基类。
2. 实现 `vllm_engine.py`：
   - `start(node_id, config)` 调用目标节点的 `POST /agent/process/start`
   - 命令模板变量化：`--model {model_path}`, `--tensor-parallel-size {tp}`, `--max-model-len {max_len}`
   - 实时日志由 Agent 写入 `{LOGS_BASE}/{workspace_id}/{deployment_id}/deploy.log`。`LOGS_BASE` 默认 `/var/log/navima`，通过节点 Agent 的环境变量 `LOGS_BASE` 配置（Ansible 下发时注入，默认值在 `scripts/agents/node-agent/install.sh` 中定义）。
3. **Agent 部署与运维规范**：
   - **安装方式**：Ansible Playbook 自动下发 `scripts/agents/node-agent/` 目录到目标节点的 `/opt/navima/agent/`，并注册为 systemd 服务（Linux）或 launchd 服务（macOS）。Windows 节点使用 nssm 包装为 Windows Service。
   - **依赖管理**：Agent 目录内含 `requirements.txt`（`fastapi`, `uvicorn`, `psutil`），Ansible 在目标节点创建独立 venv 并安装依赖。
   - **更新策略**：重新探活（probe）时，Ansible 对比本地 Agent 版本与主平台版本，不一致则自动覆盖更新并重启服务。
4. 单元测试：Mock Agent HTTP 接口，验证请求 payload 正确。
4. `xinference_engine.py` 和 `sglang_engine.py` 同理实现。

- MVP 阶段，GPU 切片层是**调度管理层**，不是驱动层。

- 所有厂商差异封装在 `gpu/slicers/` 中，上层接口统一。

#### 交互流程（组织架构同步）

1. 管理员在 SSO 配置保存时，若选择"自动同步组织架构"，系统记录 `auto_sync_org=true`。

4. 同步规则与冲突解决：

| IdP 侧变化 | 本地行为 | 冲突处理 |
|---|---|---|
| 新增部门 | 创建同名 `workspace` | 若名称已存在，跳过（不做重复创建） |
| 删除部门 | `workspace` 标记 `archived=true`，不删除数据 | 管理员手动确认后才可最终删除 |
| 用户从部门移除 | 从对应 `workspace_members` 中移除 | 若用户是 owner，先转移 ownership 给部门内 admin |
| 新增用户 | 创建 `users` 记录，`is_active=true` | 若邮箱已存在，合并 SSO 身份（更新 `sso_provider`, `sso_subject`） |
| 删除用户 | 标记 `is_active=false` | 历史数据保留 |
| 用户信息更新 | 更新 `username` 和部门归属 | 邮箱字段由 IdP 变更时，更新并发送通知 |

5. 字段映射规则（可配置）：
   - IdP `department` / `orgUnit` → `workspaces.name`
   - IdP `email` → `users.email`
   - IdP `name` / `displayName` → `users.username`
   - 无邮箱时 fallback：`{idp_username}@{DEFAULT_DOMAIN}`，仍无则 `{phone}@{DEFAULT_DOMAIN}`

## 3.5 跨模块工程规范补充（ reviewer 反馈修复）

### A. Celery 任务重试与幂等性策略

所有通过 Worker 执行的异步任务必须配置以下默认策略：

```python
# backend/worker/app/celery_config.py
task_annotations = {
    '*': {
        'max_retries': 3,
        'default_retry_delay': 10,  # 秒
        'retry_backoff': True,
        'retry_backoff_max': 300,
    }
}
```

- **幂等性**：所有任务函数接收 `task_id` 作为参数，执行前先查 `task_results` 缓存（Redis），若已有成功记录则直接返回。
- **优雅关闭**：Celery Worker 配置 `worker_shutdown_timeout = 600`（10 分钟窗口）。收到 SIGTERM 后，进行中的训练/部署任务在超时前继续执行；若超时仍未完成，任务被标记为 `interrupted`，前端提示"任务被中断，请重试"。
- **死信队列**：配置 `celery` dead-letter exchange，重试耗尽后写入 `dlq.*` 队列，触发告警。

### B. 节点凭证加密规范

`nodes.credentials_encrypted` 字段使用 **AES-256-GCM** 加密：
- 密钥来源：环境变量 `CREDENTIALS_ENCRYPTION_KEY`（32 字节 Base64）。
- 生产环境建议：通过 Kubernetes Secrets 或外部 KMS 注入，禁止写入 `values.yaml` 明文。
- 加密内容：JSON 序列化后的凭证对象（密码 / SSH Key / 证书内容）。

### C. 监控数据采集策略（MVP）

MVP 阶段不强制依赖 Prometheus，采用**自定义 Node Agent 轻量采集**：
- Agent 在每个节点上运行（随节点纳管时 Ansible 自动下发）。
- Agent API：`GET /agent/metrics` 返回 JSON（CPU、内存、GPU 利用率、显存占用）。
- `navima-api` 定期（30s）轮询各节点的 Agent API，聚合后缓存到 Redis，供监控面板查询。
- Prometheus / Grafana 集成标记为**可选增强项**（P1 之后）。

### D. 审计日志保留与分区策略

```sql
-- audit_logs 按月分区
CREATE TABLE audit_logs (
    id BIGSERIAL,
    user_id UUID REFERENCES users(id) ON DELETE SET NULL,
    action VARCHAR(100) NOT NULL,
    resource_type VARCHAR(50) NOT NULL,
    resource_id VARCHAR(100),
    detail JSONB,
    ip_address INET,
    user_agent TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (id, created_at)
) PARTITION BY RANGE (created_at);

-- 默认保留 90 天在线数据，过期后归档到对象存储（每日 Cron 任务）
```

### F. Token 与临时数据清理策略

- **verification_tokens**：Celery Beat 每小时执行清理任务，删除 `expires_at < NOW()` 或 (`used_at IS NOT NULL` 且 `created_at < NOW() - INTERVAL '1 day'`) 的记录。
- **session 缓存**：Redis 中的 session 数据通过 TTL 自动过期，无需额外清理。

### G. 数据库连接池配置

`backend/api/app/db/session.py` 创建引擎时必须显式传入连接池参数：

```python
engine = create_async_engine(
    settings.db_url,
    pool_size=settings.db_pool_size,
    max_overflow=settings.db_max_overflow,
    pool_timeout=settings.db_pool_timeout,
)
```

### H. Forward 模块超时与请求限制

`backend/api/app/modules/forward/services.py` 中配置上游调用限制：
- `FORWARD_CONNECT_TIMEOUT`：5 秒
- `FORWARD_READ_TIMEOUT`：120 秒
- `FORWARD_MAX_REQUEST_SIZE`：10 MB（通过 Nginx / Ingress 层同步限制）
- 熔断策略标记为 **P1 增强项**（MVP 不做，但接口设计需预留配置字段）。

### I. Agent Secret 轮换策略

- **自动轮换**：每次节点探活（`POST /api/v1/nodes/{id}/probe`）成功后，生成新的 `agent_secret`，通过 Ansible 更新节点 Agent 环境变量，并回写 `nodes.agent_secret`。
- **手动轮换**：提供 `POST /api/v1/nodes/{id}/rotate-secret` API，供管理员紧急轮换。
- **兼容性**：Agent 启动时读取环境变量，旧连接在 Agent 重启后失效。

### J. OpenAPI 文档策略

- `backend/api/app/main.py` 默认启用 `docs_url="/docs"` 和 `redoc_url="/redoc"`。
- 生产环境通过 `ENABLE_API_DOCS=false` 关闭文档入口（由 `config.py` 控制）。

### E. 备份与灾难恢复（最小 viable）

- **数据库**：每日凌晨 2 点执行 `pg_dumpall`，压缩后上传到对象存储（`backup/db/`）。
- **对象存储**：关键 Bucket 开启版本控制或跨区域复制（由用户环境的底层存储提供）。
- **License**：`licenses` 表和 `cluster_id` 必须包含在备份中，恢复时需校验一致性。
- **恢复文档**：在 `docs/operations/disaster-recovery.md` 中提供标准恢复流程（模块 25 后补齐）。
